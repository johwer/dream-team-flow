### Lean by design Рђћ cut your AI spend, not your output

## How It Works

```
Ticket Рєњ Architect Рєњ Parallel Dev Рєњ Code Review Рєњ Test Рєњ PR Рєњ Human Review Рєњ Ship
```

1. **You say:** [`/create-stories`](docs/commands.md#create-stories) `PROJ-1234 PROJ-1235`
2. **Dream Team does:**
   - Fetches all tickets from Jira and **pre-analyzes them in parallel** (scope, complexity, key files, conventions)
   - Presents a recommendations table Рђћ you choose Dream Team, Lite, or Just Worktree per ticket
   - Creates git worktrees, installs deps, writes pre-hydrated context files
   - Opens terminals with Claude Code sessions that start with full context
   - Spawns only the agents each ticket actually needs
   - Implements backend and frontend in parallel using a shared API contract
   - Creates a draft PR so the team can follow progress from the start
   - Runs a deterministic quality gate script (formatting, linting, type checks) before every push
   - Reviews the code for security (OWASP) and conventions
   - Polls AI bots (Gemini, Copilot) and CI Рђћ fixes issues with a 2-round cap (escalates to user after)
   - PR stays **draft** until you explicitly confirm Рђћ then marks ready and assigns reviewers
   - Moves ticket to Done and cleans up

## Token-Efficient by Design

Most AI coding setups grow expensive fast Рђћ MCP servers, large tool schemas, and bloated context windows add up quickly. Dream Team Flow is built differently:

- **No MCP servers** Рђћ agents use plain file reads and CLI tools instead of context-heavy server integrations
- **Pre-hydrated context** Рђћ `/create-stories` pre-analyzes tickets in parallel before launching sessions; agents start with scope, key files, and conventions already determined Рђћ no wasted exploration tokens
- **Deterministic nodes** Рђћ formatting, linting, type checks, and builds run as shell scripts (`quality-gate.sh`), not as LLM-reasoned steps
- **Disk-based memory** Рђћ agents write decisions and findings to `.dream-team/notes/` on disk and read them back when needed, rather than keeping everything in the context window
- **Targeted reads** Рђћ agents use Grep to find what they need rather than loading entire files or docs
- **Lite mode** Рђћ for smaller tickets, Claude skips agent spawning entirely and works solo, with no team coordination overhead
- **CI iteration cap** Рђћ 2 fix rounds max before escalating to user, preventing infinite token-burning retry loops
- **Dynamic team sizing** Рђћ the architect only spawns agents the ticket actually needs; extra agents are never started

The result: multi-agent sessions that stay lean even on large tickets.
