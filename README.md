**Free, open-source multi-agent orchestration for [Claude Code](https://docs.anthropic.com/en/docs/claude-code). Give it a Jira ticket ‚ÄĒ get back a production-ready, security-scanned, human-reviewed PR.**

Run tickets in parallel with isolated worktrees and Docker ‚ÄĒ multiply your team's story output without multiplying headcount. Keep AI costs predictable with shell-based quality gates, disk-based memory, and agents that read only what they need ‚ÄĒ no MCP servers, no bloated context windows. Onboard new developers in minutes with one install command and shared config. Ship every PR through a 7-category OWASP security scan ‚ÄĒ built into the pipeline, not bolted on after. Every bug, review comment, and best-practice article feeds back into agent prompts and coding style docs ‚ÄĒ the same mistake never happens twice.

Built on Claude Code's native multi-agent architecture ‚ÄĒ subagents, hooks, task coordination, structured tool use ‚ÄĒ following Anthropic's official patterns. No wrappers, no middleware, no vendor lock-in. When Claude Code ships a new feature, Dream Team Flow uses it directly.

### [Self-learning ‚ÄĒ your standards, your pitfalls, your codebase](#adapt-to-your-codebase)

Every bug found, every review comment, every best-practice article your team reads gets baked into agent prompts, coding style docs, and pre-hydrated context. The same mistake never happens twice. Standards stay consistent across every developer and every ticket ‚ÄĒ human or AI.

Agents that just know "React + .NET" produce generic code. Agents that know **your** serialization gotchas, **your** idempotency rules, **your** permission model produce code that passes review on the first try.

### [Parallel everything ‚ÄĒ 4x story output or more](docs/parallel.md)

Run 4, 6, 10 tickets simultaneously. Each ticket gets its own git worktree, its own Docker containers with isolated ports, and its own agent team ‚ÄĒ completely independent, zero conflicts. Backend and frontend agents work in parallel within each ticket via a shared API contract. Pause overnight, resume the next day from disk ‚ÄĒ zero token cost between sessions.

A single developer can sustain the story output of a 4-person team ‚ÄĒ or more ‚ÄĒ because the work happens concurrently, not sequentially.

### [Lean by design ‚ÄĒ cut your AI spend, not your output](docs/token-efficiency.md)

Each ticket is routed to the right mode ‚ÄĒ full team, lite, or just a worktree ‚ÄĒ so trivial tickets cost zero and only complex ones get the full team. Formatting, linting, and builds run as shell scripts ‚ÄĒ zero LLM tokens. Structured handoff templates eliminate agent back-and-forth. Deadlock detection catches stuck agents at 10 minutes. CI fixes capped at 2 rounds.

New in this release: **early triage checkpoint** kills tickets before pre-hydration tokens are spent ‚ÄĒ the user sees full Jira data and decides GO/SKIP/WORKTREE before any Sonnet agent runs. Pre-hydration agents are budget-capped at 30 tool uses. `MAX_THINKING_TOKENS` caps hidden reasoning cost (~70% reduction). `cost-tracker.sh` reports session costs with per-tool breakdowns ‚ÄĒ know exactly where tokens go. `phase-cost-tracker.sh` tracks costs per Dream Team phase so you can compare sessions and spot regressions.

Per-ticket API costs stay predictable ‚ÄĒ and now measurable.

### [Built for teams ‚ÄĒ onboard in minutes, not days](docs/built-for-teams.md)

One command to install, one command to update. `company-config.json` auto-configures service names, Jira domain, and paths ‚ÄĒ new developers ship PRs in minutes, not days. Updates deep-merge without breaking personal config. `/ticket-scout` flags vague requirements before sprint starts. i18n ships with all languages from day one.

Every session ends with a retro that feeds improvements back into prompts and docs ‚ÄĒ the self-learning loop that makes the system [adapt to your codebase](#adapt-to-your-codebase) over time.

### [Secure by default ‚ÄĒ compliance without slowing down](SECURITY.md)

Every PR gets a 7-category OWASP security scan before it reaches a human reviewer. Schema changes require Mermaid diagrams and explicit human approval ‚ÄĒ agents cannot autonomously change your data model. Quality hooks physically prevent agents from skipping steps.

Three-tier permission ladder ‚ÄĒ personal sandbox, shared standards, team-enforced lockdown. For regulated industries, security review and change governance are built into the pipeline, not bolted on after.

### [Code review that filters its own false positives](docs/review-modes.md)

Most AI code review produces a wall of findings ‚ÄĒ some real, some noise. Developers learn to ignore it all. DTF's `/review-pr --deep` solves this with a multi-agent validation pipeline:

1. **4 parallel agents** review the PR (2x convention on Sonnet, 2x bug/security on Opus)
2. **Every finding gets independently verified** by a fresh validation agent
3. **Only confirmed issues reach you** ‚ÄĒ false positives are silently filtered out

The result: fewer findings, but the ones you see are real. Combine with `--full` for local build verification ‚ÄĒ type errors and compilation failures that no diff-only reviewer can catch.

Three depth levels to match the situation: **fast** (API-only, seconds), **full** (local builds), **deep** (multi-agent + validation). Mix them: `--deep --full` for maximum thoroughness on critical PRs.

## How It Works

```
Ticket ‚Üí Architect ‚Üí Parallel Dev ‚Üí Code Review ‚Üí Test ‚Üí PR ‚Üí Human Review ‚Üí Ship
```

1. **You say:** [`/create-stories`](docs/commands.md#create-stories) `PROJ-1234 PROJ-1235`
2. **Dream Team does:**
   - Fetches all tickets from Jira and **pre-analyzes them in parallel** (scope, complexity, key files, conventions)
   - Presents a recommendations table ‚ÄĒ you choose Dream Team, Lite, or Just Worktree per ticket
   - Creates git worktrees, installs deps, writes pre-hydrated context files
   - Opens terminals with Claude Code sessions that start with full context
   - Spawns only the agents each ticket actually needs
   - Implements backend and frontend in parallel using a shared API contract
   - Creates a draft PR so the team can follow progress from the start
   - Runs a deterministic quality gate script (formatting, linting, type checks) before every push
   - Reviews the code for security (OWASP) and conventions
   - Polls AI bots (Gemini, Copilot) and CI ‚ÄĒ fixes issues with a 2-round cap (escalates to user after)
   - PR stays **draft** until you explicitly confirm ‚ÄĒ then marks ready and assigns reviewers
   - Moves ticket to Done and cleans up

See **[Workflow Phases](docs/workflow-phases.md)** for detailed flowcharts of full, lite, and local modes.

## Features

- **Parallel context pre-hydration** ‚ÄĒ All tickets analyzed in parallel before sessions start
- **Dynamic team sizing** ‚ÄĒ Architect spawns only the agents the ticket needs
- **Draft PR from the start** ‚ÄĒ Created after architecture analysis; stays draft until user explicitly confirms
- **Drift detection** ‚ÄĒ Build baseline captured before implementation, compared before push ‚ÄĒ regressions caught locally, not in CI
- **Crash recovery** ‚ÄĒ Agents write notes to disk; crashed agents respawn with full context
- **Stale worktree cleanup** ‚ÄĒ Merged/closed PRs detected automatically; orphan worktrees cleaned up before new ones are created
- **Non-destructive PR updates** ‚ÄĒ PR description edits preserve manually added images and screenshots
- **Jira completion comments** ‚ÄĒ Ticket creator auto-notified with PR link and summary when work is done
- **Standalone PR review** ‚ÄĒ `/review-pr` reviews any PR with line-level GitHub comments. Three depth levels: fast (API-only), full (`--full` ‚ÄĒ local builds/tests), deep (`--deep` ‚ÄĒ 4 parallel agents + validation pass to eliminate false positives). Combine `--deep --full` for maximum depth
- **Granular task decomposition** ‚ÄĒ 5-6 small tasks per agent instead of 1-2 big ones ‚ÄĒ better progress visibility, checkpoint-level quality enforcement, and less work lost on crashes
- **De-sloppify pass** ‚ÄĒ Cleanup phase before commit: removes over-engineering, dead code, premature abstractions, and defensive bloat that agents naturally introduce
- **Strategic compaction** ‚ÄĒ Proactive context management with defined compact points at phase boundaries ‚ÄĒ prevents quality degradation in long sessions
- **Context modes** ‚ÄĒ Dev/review/research mindsets with distinct priorities and tool preferences ‚ÄĒ auto-activates for relevant commands
- **Continuous learning from tool usage** ‚ÄĒ 4th learning path: analyze tool usage patterns, detect context gaps and struggle points, promote to skills/conventions/scripts via `/evolve`
- **Cost tracking** ‚ÄĒ Session cost reports with relative units per tool, top sessions, and per-session breakdowns
- **Config health scanner** ‚ÄĒ Security/health scan of your Claude Code configuration (secrets, performance, hygiene) with A-F grading

See **[Features](docs/features.md)** for the full list ‚ÄĒ team setup, orchestration, review, resilience, and self-learning.

## Adapt to Your Codebase

Dream Team Flow isn't just configurable ‚ÄĒ it **learns your codebase** through six learning channels that continuously feed knowledge into agent prompts, coding style docs, and project conventions:

| Channel | Source | Example |
|---------|--------|---------|
| **Session retros** | Agents reflect after every ticket | "Handlers should use atomic upserts" |
| **PR review insights** | Patterns from merged PR feedback | "6 PRs had missing i18n ‚ÄĒ add to quality gate" |
| **Jira pushback scraping** | AI reviewer comment analysis | "Tickets without ACs get 3x more review cycles" |
| **Tool usage patterns** | Behavioral analysis from tool logs | "settings.json read 12x ‚ÄĒ add to CLAUDE.md" |
| **Research & reading** | Internet articles, docs, best-practice guides | Microservices messaging patterns from an online guide |
| **Cross-session analysis** | Pattern detection across retros | "4 sessions mentioned soft delete confusion" |

Every learning flows to the right destination ‚ÄĒ agent prompts, coding style docs, pre-hydrated context, or command logic. Both humans and AI agents read the same docs, so knowledge compounds for everyone.

**Example:** Your team reads an article on microservice messaging and extracts key patterns ‚ÄĒ idempotent handlers, no sync calls from event handlers, transactional outbox. Those learnings become:
- A new section in your backend coding style guide (repo docs, via Jira + PR)
- A critical rule in the backend agent's prompt ("every handler MUST be idempotent")
- Pre-hydrated context for any ticket touching message handlers

New team members inherit **everything** on day one ‚ÄĒ the accumulated knowledge isn't in someone's head, it's in agent prompts, coding style docs, and context files that every session reads automatically.

Read more: **[Self-Learning System](docs/retrospectives.md)** ‚ÄĒ all five channels, routing rules, destinations, and the compounding effect.
